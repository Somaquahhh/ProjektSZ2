Studio13 – fejlesztői dokumentáció

Ez az oldal a projekt működését, felépítését, jogosultságait, API-ját és adatbázisát írja le.
Gyors összefoglaló
  • Backend: Node.js + Express (v5), express-session, MySQL (mysql driver).
  • Frontend: statikus HTML oldalak Bootstrap + jQuery alapon, közös segédfájl: common_studio13.js.
  • Auth: session cookie (1 óra). Oldalak és API végpontok bejelentkezéshez kötöttek.
  • Szerepkörök: Students, Teachers, Boss. Oldal-hozzáférés és menü ezek szerint szűrődik.

1. Architektúra

  • Szerver belépési pont: std13.js (Express app).
  • DB modul: datamodule_mysql.js – pool alapú MySQL kapcsolat + egységes JSON válaszformátum.
  • Statikus tartalom: public/ alatt HTML/CSS/JS (az Express innen szolgálja ki).
Fő adatfolyam
  1. Felhasználó belép a /login.html oldalon → POST /login
  2. Sikeres login esetén session létrejön (express-session)
  3. Az oldalak többsége csak sessionnel elérhető (HTML guard + API middleware)
  4. Frontend fetch/$.ajax hívásokkal éri el a /api/* végpontokat

2. Indítás / fejlesztői környezet

Függőségek
  • express, express-session, mysql (lásd: package.json)
  • Node.js ajánlott: >= 18 (a lockfile szerint)
Telepítés
npm install
Indítás
A repóban a package.json alapból nem tartalmaz start scriptet; a szerver belépési pont a std13.js.
node std13.js
# ezután: http://localhost:9021
Könyvtár-struktúra (logikai)

├─ package.json
└─ public/
   ├─ index.html

3. Jogosultságok és védelem

Szerepkörök
  • Students – alap jogosultság
  • Teachers – tanári funkciók
  • Boss – admin / napló funkciók
Oldal-hozzáférés (PAGE_MIN_ROLE)
  • Students: /index.html, /sajat.html, /b.html
  • Teachers: /a.html, /rfid.html, /rfidki.html, /keseses.html
  • Boss: /admin.html, /naplozos.html
  • Publikus: /login.html
Middleware logika röviden
  • authMiddleware: ha nincs session → API-nál 401, HTML-nél redirect /login.html.
  • guardHtmlByRole: HTML oldalaknál role-check; Boss mindent láthat, egyébként minimum szerepkör kell.
  • requireBoss: /api/admin/* csak Boss.
Session adatok
// sikeres login után a sessionben:
req.session.ID_USER
req.session.NEV
req.session.EMAIL
req.session.CSOPORT   // normalizálva (Bosses → Boss, trailing ';' levágva)
req.session.OM
req.session.MOST      // timestamp

4. Oldalak (public)

Fő oldalak
  • login.html – belépés, POST /login, session ellenőrzés: GET /session.
  • index.html – fő keret, menü + AJAX betöltés #main1-be (a menü pontok több oldalt fragmentként töltenek be).
  • sajat.html – saját session adatok + saját kibe események listája: GET /api/my/kibe.
Tanári / admin oldalak
  • a.html – adatkereső (users lista) → POST /asd.
  • b.html – Kikérő: diák beküld / tanár elbírál. API: /api/kikero*, /api/tanarok.
  • rfid.html – beléptetés (direction=in), API: GET /api/rfid/event.
  • rfidki.html – kiléptetés (direction=out), API: GET /api/rfid/event.
  • keseses.html – késések listája + szűrők, API: POST /api/kibe_simple.
  • admin.html – user CRUD (Boss), API: /api/admin/users.
  • naplozos.html – admin napló (Boss), API: /api/admin/kibe, /api/admin/kikero.

5. API dokumentáció

Mindegyik API endpoint sessionhöz kötött Kivéve: /login, /session
5.1 Auth
POST /login
Body: { user, psw } (user = email VAGY név). Jelszó ellenőrzés: MD5(psw).
// request
POST /login
Content-Type: application/json


{ "user": "valaki@pelda.hu", "psw": "titok" }


// response (DB wrapper formátum)
{ "count": 1, "rows": [ { "ID_USER": 1, "NEV": "...", "CSOPORT": "Students", "EMAIL": "...", "OM": "..." } ], "tip": "info", "text": 0 }
GET /session
// ha nincs login:
{ "bejelentkezett": false }


// ha van login:
{
  "bejelentkezett": true,
  "user": { "ID_USER": 1, "NEV": "...", "EMAIL": "...", "CSOPORT": "Students", "OM": "..." }
}
POST /logout
// response:
"Sikeres kijelentkezés."
GET /api/me
Gyors role-check (admin felületek használják).
{ "id": 1, "role": "Boss" }
5.2 Tanárok
GET /api/tanarok
Tanári kikérőnél a választólistához.
{ "count": 12, "rows": [ { "id": 7, "nev": "Név" }, ... ] }

5.4 RFID + kibe modul
GET /api/rfid
RFID kódhoz tartozó user gyors lekérése.
/api/rfid?kod=RFID123


// response
{ "ID_USER": 1, "NEV": "...", "EMAIL": "...", "OM": "..." }
GET /api/rfid/event
RFID esemény rögzítése a kibe táblába.
/api/rfid/event?
  kod=RFID123
  &direction=in|out
  &graceHours=0
  &start=07:50
  &end=16:00


// response (példa)
{
  "ID_USER": 1,
  "NEV": "...",
  "EMAIL": "...",
  "OM": "...",
  "saved": true,
  "direction": "in",
  "graceHours": 0,
  "shift": { "start": "07:50", "end": "16:00" },
  "status": "ok|late|early_leave",
  "diffMinutes": 7,
  "kesesPerc": 7
}
Megjegyzés: késés perc csak belépésnél (direction=in) kerül mentésre, kilépésnél 0.
GET /api/my/kibe
Saját be/ki események (dátum szerint csökkenő).
[ { "ID_KIBE": 1, "DIRECTION": "in", "RFID_POZ": "RFID123", "DATUMIDO": "..." }, ... ]
GET /api/minden_keses
Összes késés (csak azok, ahol KESES_PERC > 0).
[ { "NEV": "...", "OM": "...", "DATUMIDO": "...", "KESES_PERC": 5 }, ... ]
POST /api/kibe_simple
Kompatibilitási endpoint: STRING JSON-t küld vissza (keseses.html használja). Limit: 1000 sor.
// response: "{\"count\":...,\"rows\":[...]}"  (string)
// vagy a frontend oldalon JSON.parse után:
{ "rows": [ { "ID_KIBE":..., "NEV":..., "DATUMIDO":..., "DIRECTION":..., "RFID_POZ":..., "KESES_PERC":... }, ... ] }

6. Adatbázis (táblák és mezők)

6.1 users
Felhasználók táblája. Bejelentkezés: EMAIL vagy NEV + MD5(PASSWORD). RFID azonosítás: RFID mező alapján.
  • ID_USER (PK)
  • LOGIN (admin felületen létrehozáskor kötelező)
  • NEV
  • EMAIL
  • CSOPORT (Students | Teachers | Boss – előfordulhat Bosses vagy a végén ';', a backend normalizálja)
  • PASSWORD (MD5 hash)
  • OM (string/number; admin update-ben 11 számjegy ellenőrzés van)
  • RFID (RFID kód; /api/rfid és /api/rfid/event ezt használja)
  • DATUMIDO (timestamp; user létrehozáskor NOW())
6.2 kibe
RFID belépés/kilépés események táblája. Az /api/rfid/event ide szúr be rekordot.
  • ID_KIBE (PK)
  • ID_USER (FK → users.ID_USER)
  • DATUMIDO (timestamp; jellemzően DEFAULT NOW())
  • DIRECTION ('in' | 'out')
  • RFID_POZ (a leolvasott RFID kód)
  • KESES_PERC (int; csak direction='in' esetén töltjük, különben 0)
6.3 kikero
Kikérő kérések táblája (diák → tanár). A tanár az állapotot /api/kikero/allapot-tal állítja.
  • ID_KIKERO (PK)
  • ID_KERO (FK → users.ID_USER, a diák)
  • ID_TANAR (FK → users.ID_USER, a tanár)
  • NAP (date / varchar)
  • OK (varchar)
  • RESZLETEK (text, nullable)
  • ALLAPOT (UJ | ELFOGADVA | ELUTASITVA; tipikusan default 'UJ')
  • LETREHOZVA (timestamp; tipikusan DEFAULT NOW())
  • DONTES_IDO (timestamp, nullable)
  • DONTES_MEGJEGYZES (text, nullable)
6.4 naplo_audit
Audit napló tábla. A backend auditLog() helperrel ide ír (belépés, kijelentkezés, admin törlés/módosítás/létrehozás).
  • ID_NAPLO (PK)
  • DATUM_IDO (timestamp; tipikusan DEFAULT NOW())
  • FELHASZNALO_ID (int, nullable; sessionből)
  • FELHASZNALO_NEV (string, nullable; sessionből)
  • MUVELET (string; pl. BELEPES, KIJELENTKEZES, TORLES, MODOSITAS, LETREHOZAS)
  • OBJEKTUM (string; pl. auth, users, kibe, kikero)
  • OBJEKTUM_ID (int/string, nullable; pl. érintett rekord ID-ja)
  • RESZLETEK (text, nullable; extra infó JSON/string formában)
  • IP_CIM (string; kérés IP)

7. Frontend közös logika (common_studio13.js)

Funkciók
  • Role-alapú menüszűrés: /session alapján állítja be a USER_ROLE-t, és csak a megfelelő menüpontokat rendereli.
  • Menü generátor: menü_json → #menu1_ul elemek.
  • AJAX HTML betöltés: ajax_get(url, "#main1", 0, true). Külön kezeli a <script> tageket (dupla betöltés ellen cache-eli a külső script src-eket).
  • Toast üzenetek: mySend({text, tip, mp}) (Bootstrap Toast).
  • Oldalfüggő súgó: jobb alsó sarok ❓ Súgó gomb; AJAX esetben is felismeri az aktuális oldalt.
AJAX + script kezelés miért fontos?
Az index.html keretből több oldal fragmentként töltődik be a #main1-be. Ilyenkor a DOM-ba bekerülő inline script-eket kontrolláltan kell futtatni, és a külső script-eket nem szabad többször betölteni.

8. Hibakeresés

Gyakori hibák
  • 401 / 403 API-n: nincs session vagy nincs jogosultság. Ellenőrizd: GET /session és role.
  • Modal eltűnik AJAX betöltésnél: a.html már kezeli (modal áthelyezése document.body-ba megnyitás előtt).
  • keseses.html nem tölt adatot: az endpoint /api/kibe_simple POST; ellenőrizd a sessiont.
  • RFID event hibák: rossz HH:MM formátum (start/end), hiányzó kod, rossz direction.
